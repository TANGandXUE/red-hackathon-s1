# HinH - Hackathon in Hackathon 设计文档

## 概述

HinH 是一个模拟沙盘产品，用户输入 idea 后，20 个 AI 驱动的 OC 角色在像素风酒馆场景中模拟黑客松全流程（讨论→创造→答辩），最终产出结构化 BP 文档和评委评分。

核心价值：多角色多视角推演，将粗略想法打磨为有商业价值的成熟方案。

## 业务流程

### 入口：Idea 输入页

- 全屏像素风界面，标题"欢迎来到 HinH"
- 支持输入 1-4 个 idea
- 1 个 idea → 同一 idea 分发给 4 组各自推演
- 多个 idea → 随机分配给不同组
- 点击"开始模拟"进入酒馆场景

### 阶段 0：分组（自动，无对话）

- 20 个 OC 角色按性格特征匹配岗位（产品经理、前端工程师、后端工程师、设计师、运营）
- 分成 4 组，每组 5 人，选出队长
- 地图上角色走动入座到各自桌子

### 阶段 1：自由讨论（4 组并行）

每组内流程：
1. 队长收到 idea，发言提出方向
2. 组员依次各提一个问题
3. 每人自由选 1-2 条已有问题进行回答或反问
4. 队长总结敲定选题和产品方向

### 阶段 2：创造产品（4 组并行）

每组内流程：
1. 队长分配任务
2. 组员依次确认/提问
3. 队长确认分工
4. 各组员撰写负责的 BP 板块（可联网搜索）
5. 队长汇总生成完整 BP 文档

BP 结构（精简版）：项目名称、问题与痛点、解决方案、目标用户、核心功能、商业模式、竞争优势

### 阶段 3：评审答辩（逐组进行）

场景切换为答辩厅（上排 5 选手席，下排 6 评委席）。每组：
1. 上台动画，队长居中
2. 队长陈述 BP
3. 2-3 个评委提问（AI 决定谁提问）
4. 队长或相关岗位组员回答（AI 决定谁回答）
5. 6 评委打分 + 点评 + 改进建议

评分维度（每项 1-10 分）：创新性、现场讲述效果、完成度、商业潜力、技术难度

### 结果页

- 各组排名（按总分）
- 每组可展开查看：BP 文档、评委点评、改进建议

## 角色体系

### OC 角色（20 个）

- 7 个来自问卷数据（含像素风 OC 头像）
- 13 个系统预生成的虚拟角色（开发时一次性生成写死）
- 属性：名字、形态、色调、性格（MBTI）、描述、技能、武器、装备、伙伴

产品经理、前端工程师、后端工程师、设计师、运营

## 技术栈

- 前端：Next.js 16 + TypeScript + Tailwind CSS 4 + shadcn/ui + Zustand + Phaser 3
- 后端：NestJS 11 + TypeORM + PostgreSQL
- LLM：Claude Haiku 4.5 via api.bltcy.ai（OpenAI 兼容格式）
- 实时通信：SSE
- 联网搜索：全员可用，全局队列控制并发（max 3）

## 前端结构

| 页面 | 路由 | 说明 |
|---|---|---|
| Idea 输入页 | `/` | 全屏像素风，输入 idea |
| 模拟页 | `/simulation` | 左 Phaser 酒馆/答辩厅 + 右对话 Tab |
| 结果页 | `/result` | 排行榜 + BP + 评委点评 |

- Phaser 3 嵌入 React 组件，通过 Zustand store 做状态同步
- 使用 Phaser 的 Sprite、Tween、Tilemap（不用物理引擎）
- 右侧 4 个 Tab 切换各组对话

## 后端架构

```
SimulationOrchestrator
├── LLMClient（Semaphore 限流）
├── SearchQueue（并发 3，结果缓存）
├── GroupRunner x4（并行）
│   ├── agents: AgentInstance x5（persona prompt + 对话历史）
│   └── phaseExecutor: 按阶段规则驱动发言顺序
├── PhaseGate（全组完成 → 下一阶段）
├── JudgeRunner（逐组答辩）
└── SSE 推送
```

每个 Agent = persona prompt + 本组对话历史 → 调 LLM → 返回回复。无额外抽象。

## 阶段驱动规则

| 阶段 | 发言模式 |
|---|---|
| 0 | 无对话，算法分组 |
| 1 | 队长→组员轮流提问→自由回答→队长总结 |
| 2 | 队长分配→组员确认→各自产出→队长汇总 |
| 3 | 队长陈述→评委提问→组员回答→评委打分 |

## 开发原则

1. 单一职责 — 每个服务/方法只负责一个职责域
2. 最简代码 — 不做向后兼容，删除所有冗余
3. 类型严格 — 不使用 any，编译错误立即修复
4. KISS — 保持简单直接
5. 文档置信度 — 不基于推测写代码
